## Problem Statement

앱인토스(토스 미니앱) 플랫폼에 동물 캐릭터 기반 날씨앱을 만든다. 핵심 컨셉은 **"레퍼런스 오버로드"** — 소설, 동화, 속담, 영화, 전래동화 등에서 온 동물 캐릭터들이 오늘의 날씨에 맞춰 등장하고, 자신의 백스토리를 기반으로 위트 있는 한마디를 던진다.

Micro-Product Factory 전략의 일환으로, 2주 룰 안에서 런칭하되 "쓸데없는 고퀄" 일러스트로 차별화한다.

## What Makes This Cool

**"레퍼런스 오버로드"** — 날씨를 확인하는 일상적 행위에 문학적/문화적 깊이를 입힌다.

- 비 오는 날 피터래빗이 "맥그리거 아저씨 밭에는 못 가겠네"라고 하고
- 안개 낀 날 "어디선가 본 듯한 고양이"가 "우리 모두 여긴 좀 이상하지"라고 하고
- 바람 부는 날 아기돼지 삼형제의 셋째가 "벽돌집을 지어둬서 다행이야"라고 하는 것

**3가지 Whoa 모먼트:**
1. 캐릭터 + 한마디의 조합이 터지는 순간
2. 공유 카드 비주얼이 SNS에 올리고 싶을 만큼 예쁜 순간
3. 도감에서 새로운 캐릭터를 발견하는 수집의 쾌감

**차별화:** Carrot Weather는 텍스트 유머만, Weather Puppy는 실사 사진만, Google Froggy는 단일 캐릭터. 고퀄 일러스트 + 문화적 레퍼런스 + 수집 메카닉스를 합친 날씨앱은 존재하지 않음.

## Constraints

- **플랫폼**: 앱인토스 (토스 미니앱) — React 19 + Vite 8 + TypeScript + Tailwind v4
- **시간**: Micro-Product Factory 2주 룰
- **저작권**: 퍼블릭 도메인 캐릭터는 자유 사용, 현대 저작물은 "알아볼 수 있지만 저작권에 걸리지 않는" 오마주 방식으로 트윅
- **캐릭터 생성**: Recraft API → Cloudinary CDN (검증된 파이프라인)
- **날씨 데이터**: 공공데이터포털 기상청 API (한국 날씨 정확도 최고)
- **비용**: 무료 API 티어 활용 (기상청 무료, Recraft 크레딧 관리 필요)

## Premises

1. "날씨 확인"은 하루 1회 이상 반복되는 행동이고, 캐릭터가 이 루틴을 "기대되는 순간"으로 바꿀 수 있다.
2. "레퍼런스 오버로드" — 사람들이 아는 이야기의 캐릭터를 쓰면 "이거 알아!" 신호가 공유 트리거가 된다.
3. 캐릭터 일러스트의 "쓸데없는 고퀄"이 핵심 차별화이고, Recraft API로 2주 내 생성 가능하다.
4. 앱인토스 내 날씨앱 카테고리는 블루오션이다 (캐릭터 기반 날씨앱 0개 확인).
5. 실제 날씨 데이터 API 연동이 MVP에 필수다 (더미 데이터로는 날씨앱의 의미가 없음).

## Cross-Model Perspective

Claude 서브에이전트의 독립 리뷰 결과:

- **가장 쿨한 미탐색 버전 — "Weather Oracle"**: 캐릭터가 단순 날씨 코멘트가 아니라, 자기 이야기의 관점에서 오늘 하루에 대한 인생 조언을 던지는 것. "날씨 확인"을 "일일 오라클 상담"으로 전환. 이는 회사력 테스트의 "4단 레이어(유머→거울→코칭→액션)" 패턴과 일치.
- **캐릭터 연속성**: 어제 등장한 캐릭터를 오늘 캐릭터가 언급하는 "연속극 효과" 제안. 리텐션 극대화 가능하나 MVP에서는 스킵, v1.1에서 추가 권장.
- **핵심 인사이트**: "쓸데없는 고퀄"이라는 표현이 빌더의 본질을 드러냄 — 유틸리티가 아니라 기대를 초과하는 크래프트에 흥분하는 사람.
- **주말 빌드 플랜**: 12종→30종으로 확장 결정, 토요일 캐릭터 레지스트리+Recraft 배치, 일요일 MVP+공유카드+도감.

## Approaches Considered

### Approach A: 라이트 MVP
- 캐릭터 12종, 사전 작성 대사 5개/캐릭터
- Effort: S | Risk: Low
- Pros: 2주 내 확실한 런칭
- Cons: 캐릭터 수 부족으로 도감 쾌감 약함, 대사 반복

### Approach B: Weather Oracle (AI 대사)
- 캐릭터 12종 + AI 대사 생성 (매번 다른 한마디)
- Effort: M | Risk: Med
- Pros: 매번 새로운 경험, 10x 비전 직결
- Cons: 2주 초과 가능, AI 톤 QC 필요

### Approach C (선택): 하이브리드 + 풀 캐릭터 풀
- 캐릭터 30종 (24 표준 + 6 히든), 캐릭터 풀 구조
- 사전 작성 대사 5개/캐릭터로 런칭 → v1.1에서 AI 대사 추가
- 크로스오버 이벤트는 v1.1에서 추가
- Effort: S→M | Risk: Low
- Pros: 2주 런칭 + 풍부한 도감 + AI 확장성
- Cons: 초기 30장 일러스트 배치 생성 필요 (Recraft 크레딧 관리)

## Recommended Approach

**Approach C: 하이브리드 + 풀 캐릭터 풀**

### 핵심 구조: 캐릭터 풀

캐릭터 30종이 하나의 풀로 존재. 각 캐릭터는 등장 가능한 날씨 태그를 복수개 보유.

```
{
  id: "peter-rabbit",
  name: "피터 (어느 정원의 토끼)",     // 저작권 안전 트윅
  source: "비어트릭스 포터 오마주",
  weatherTags: ["rain", "cloudy", "fog"],
  rarity: "standard",                  // standard | hidden
  personality: "호기심 많고 무모한 낙천가",
  lines: ["맥그리거 아저씨 밭엔 못 가겠네...", ...],
  colors: { primary: "#7B9E5C", ... },
  recraftPrompt: "..."
}
```

**매칭 로직:**
1. 기상청 API에서 현재 날씨 코드 수신 (SKY, PTY, TMP, WSD, RN1, SNO)
2. **히든 체크 우선**: `checkHiddenTrigger(rawData)` — 원시 API 필드(RN1, SNO, TMP)로 히든 캐릭터 트리거 확인. 히든이 발동하면 히든 캐릭터 표시 (표준 풀 스킵)
3. **표준 매칭**: `getWeatherTag(sky, pty, tmp, wsd)` → `characters.filter(c => c.weatherTags.includes(currentTag) && c.rarity === 'standard')`
4. 매칭된 캐릭터 중 1마리 랜덤 등장 (어제와 중복 방지: `lastShown.characterId` 체크)
5. 해당 캐릭터의 대사 풀에서 1개 랜덤 선택

**히든 캐릭터 (6종):**
기상청 단기예보 API에서 실제 제공되는 값 기준으로 트리거:

| 히든 조건 | 트리거 로직 (기상청 코드 기반) |
|-----------|-------------------------------|
| 폭우 | PTY=1 (비) AND RN1 ≥ 30mm/h |
| 폭설 | PTY=3 (눈) AND SNO ≥ 5cm |
| 극한 추위 | TMP ≤ -10°C |
| 극한 더위 | TMP ≥ 35°C |
| 강풍 | WSD ≥ 21m/s (강풍주의보 수준) |
| 새벽흐림 | 06:00 이전 AND SKY=4 (흐림) AND TMP ≤ 5°C |

도감에 실루엣으로 표시 + 힌트("폭우가 쏟아지는 날 나타나요"), 발견 시 성취감 + 공유 트리거

### 저작권 전략

| 소스 | 전략 | 예시 |
|------|------|------|
| 퍼블릭 도메인 (이솝/그림/안데르센/전래동화) | 자유 사용 | "이솝의 여우", "미운 오리" |
| 고전 문학 (70년+ 경과) | 자유 사용 | "모비딕의 고래", "어린왕자의 여우" |
| 한국 전래동화/속담 | 자유 사용 | "토끼와 거북이의 토끼", "호랑이" |
| 현대 저작물 | 오마주 트윅 | "숲지기 곰" (토토로 오마주), "번개 다람쥐" |

**트윅 원칙 (구체적 가이드라인):**
1. **이름**: 반드시 오리지널 이름 사용. 원작 캐릭터명 직접 사용 금지
2. **비주얼**: 원작의 고유 시각 요소(토토로의 잎사귀 우산, 피카츄의 번개 꼬리 등) 사용 금지. 독립적으로 새로 디자인
3. **성격/톤**: 원작의 분위기를 연상시키는 정도는 OK. 특정 대사 직접 인용은 금지
4. **법적 안전**: 현대 오마주 캐릭터는 4-6종으로 제한. 나머지 24종은 퍼블릭 도메인에서 충당
5. "어디서 본 것 같은데...?" 가 웃음 포인트

같은 캐릭터도 날씨 강도/시간에 따라 표정·포즈·소품이 달라질 수 있음.

### 기상청 API 날씨코드 → weatherTag 매핑

```typescript
// 기상청 단기예보 카테고리 코드
// SKY: 1=맑음, 3=구름많음, 4=흐림
// PTY: 0=없음, 1=비, 2=비/눈, 3=눈, 4=소나기

function getWeatherTag(sky: number, pty: number, tmp: number, wsd: number): WeatherTag {
  if (pty === 3 || pty === 2) return 'snow';
  if (pty === 1 || pty === 4) return 'rain';
  if (wsd >= 14) return 'wind';          // 강풍 14m/s 이상
  if (sky === 1 && tmp >= 33) return 'hot';
  if (sky === 1 && tmp <= -5) return 'cold';
  if (sky === 1) return 'clear';
  if (sky === 3) return 'partly_cloudy';
  if (sky === 4) return 'cloudy';
  return 'clear'; // fallback
}

type WeatherTag = 'clear' | 'partly_cloudy' | 'cloudy' | 'rain' | 'snow' | 'wind' | 'hot' | 'cold';
```

### 기상청 API 연동 주의사항

- **좌표 변환 필수**: Geolocation 위경도 → 기상청 LCC 격자 좌표 (Lambert Conformal Conic). 변환 공식 구현 또는 `lamcproj` 라이브러리 활용
- **base_time 규칙**: 0200/0500/0800/1100/1400/1700/2000/2300 (3시간 간격). 현재 시각 기준 가장 최근 base_time 계산 필요. API 발표 후 약 10분 딜레이 (base_time 0200이면 02:10부터 조회 가능)
- **응답 파싱**: JSON 내 `response.body.items.item[]` 배열에서 카테고리별(TMP, SKY, PTY, POP, WSD 등) 필터링
- **API 키**: 공공데이터포털 회원가입 → "기상청 단기예보 조회서비스" 활용 신청 (즉시 발급)
- **일일 트래픽**: 10,000콜/일 (무료). 앱인토스 사용자 규모에 충분

### localStorage 스키마

```typescript
interface WeatherAnimalStorage {
  // 도감 진행도
  discoveredCharacters: {
    [characterId: string]: {
      firstSeen: string;     // ISO date
      seenCount: number;
      lastWeather: WeatherTag;
    };
  };
  // 마지막 표시 캐릭터 (중복 방지용)
  lastShown: {
    characterId: string;
    date: string;            // YYYY-MM-DD
    line: string;
  };
  // 사용자 위치 캐시
  locationCache: {
    lat: number;
    lng: number;
    gridX: number;           // 기상청 격자 X
    gridY: number;           // 기상청 격자 Y
    cityName: string;
    updatedAt: string;
  } | null;
}
```

### 에러 상태 & 엣지 케이스

| 상황 | 처리 |
|------|------|
| **위치 권한 거부** | 서울(기본 도시) 날씨 표시 + "위치를 허용하면 동네 날씨를 알려줄게요" 안내 |
| **기상청 API 장애/타임아웃** | 마지막 캐시된 날씨 데이터 사용 + "조금 전 날씨예요" 표시 |
| **매칭 캐릭터 0마리** | 절대 발생하지 않도록 모든 캐릭터 중 1종을 `fallback: true`로 지정 (모든 날씨에 등장 가능) |
| **오프라인** | 마지막 캐릭터+날씨 캐시 표시 + "인터넷 연결을 확인해주세요" |

## What I noticed about how you think

- "쓸데없는 고퀄이어야 한다" — 유틸리티보다 크래프트에서 에너지를 얻는 빌더. 기대를 초과하는 퀄리티가 "왜 이렇게까지?" 반응을 만들고, 그게 바이럴의 핵심이라는 걸 직감적으로 안다.
- "레퍼런스 오버로드"라는 네이밍 자체가 콘텐츠 전략을 한 단어로 압축한 것. 기능이 아니라 **경험의 밀도**로 차별화하려는 사고.
- "같은 캐릭터여도 상황에 따라 배리에이션" — 한 번 보고 끝이 아니라 "오늘은 뭐가 나올까"를 만드는 리텐션 설계를 자연스럽게 떠올림. 게이미피케이션 감각.
- 저작권 문제를 "알아볼 수 있지만 걸리지 않는 교묘한 방식"으로 해결하겠다는 것 — 제약을 오히려 위트 레이어로 전환하는 창의적 사고. "어디서 본 것 같은데...?"가 웃음이 되는 구조.
